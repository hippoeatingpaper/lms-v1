## 5-1: 팀 과제 제출 로직

### 팀원 제출 권한 테스트
- [ ] 팀원 A가 팀 과제 제출 생성 가능
- [ ] 팀원 B가 동일 팀 과제 제출물 수정 가능
- [ ] 팀원 C가 동일 팀 과제 제출물 조회 가능
- [ ] 팀에 속하지 않은 학생은 접근 불가

### 제출 충돌 방지 테스트
- [ ] 제출물에 version 필드 존재
- [ ] 수정 시 version 검증
- [ ] 동시 수정 시도 시 충돌 에러 반환
- [ ] 충돌 시 적절한 에러 메시지 (409 Conflict)

#### 충돌 시나리오 테스트
```
1. 팀원 A가 제출물 조회 (version: 1)
2. 팀원 B가 제출물 조회 (version: 1)
3. 팀원 A가 수정 완료 (version: 2로 업데이트)
4. 팀원 B가 수정 시도 (version: 1 기준) → 충돌 에러
5. 팀원 B가 최신 데이터 조회 후 재시도 → 성공
```
- [ ] 위 시나리오가 정상 동작

### 팀 제출 현황 조회 테스트
- [ ] 교사: 팀별 제출 현황 조회 가능
- [ ] 각 팀의 제출 상태 표시 (미제출/제출/평가완료)
- [ ] 제출한 팀원 정보 표시
- [ ] 제출 시간 표시

### 최종 제출 테스트
- [ ] 팀원 누구나 최종 제출 가능
- [ ] 최종 제출 후 모든 팀원 수정 불가
- [ ] 최종 제출 취소는 교사만 가능

### 피드백 테스트
- [ ] 팀 과제 피드백은 팀 전체에 적용
- [ ] 모든 팀원이 피드백 조회 가능
- [ ] 피드백 알림은 팀원 전체에게 발송

## 통합 테스트 시나리오

### 팀 과제 완전 플로우
```
1. 교사: 팀 과제 출제
2. 팀원 A: 제출물 생성
3. 팀원 B: 같은 제출물 수정 (version 검증)
4. 팀원 A: 추가 수정 (version 충돌 발생)
5. 팀원 A: 최신 버전 조회 후 재시도
6. 팀원 B: 최종 제출
7. 팀원 A: 수정 시도 → 불가
8. 교사: 피드백 작성
9. 팀원 A, B 모두 피드백 확인
```
- [ ] 위 플로우가 정상 동작

### 오프라인 시나리오
```
1. 학생이 과제 상세 페이지 열기
2. 네트워크 끊김
3. 오프라인 배너 표시 확인
4. 답안 작성 시도 → 저장 실패 메시지
5. 네트워크 복구
6. 배너 사라짐 확인
7. 답안 저장 성공
```
- [ ] 위 시나리오가 정상 동작

### PWA 설치 시나리오

1. 모바일 Chrome에서 사이트 접속
2. 일정 조건 충족 후 설치 배너 표시
3. 설치 진행
4. 홈 화면에 앱 아이콘 추가 확인
5. 아이콘으로 앱 실행
6. 독립 창에서 앱 동작 확인
7. 오프라인에서 기본 화면 로드 확인

## 버전 관리 테스트 (팀 제출)

### 낙관적 잠금 동작
```
현재 버전: 1
수정 요청 시:
  - 요청 version === DB version → 수정 성공, version++
  - 요청 version !== DB version → 409 Conflict
```

### API 응답 형식
```json
// 성공
{
  "id": "sub_123",
  "content": "...",
  "version": 2
}

// 충돌
{
  "error": {
    "code": "VERSION_CONFLICT",
    "message": "다른 팀원이 먼저 수정했습니다. 새로고침 후 다시 시도해주세요.",
    "currentVersion": 2
  }
}
```
- [ ] 응답 형식이 위와 일치

## 주의사항

1. **버전 충돌**: 사용자 친화적 메시지와 재시도 가이드 제공 필요
2. **오프라인 데이터**: 민감한 데이터는 로컬에 저장하지 않도록 주의
3. **PWA 캐시**: 개발 중 캐시로 인한 혼란 방지를 위해 DevTools에서 캐시 비활성화
4. **iOS 제한**: iOS Safari는 PWA 기능이 제한적 (푸시 알림 불가 등)
5. **HTTPS 필수**: PWA와 Service Worker는 HTTPS에서만 동작 (localhost 제외)
